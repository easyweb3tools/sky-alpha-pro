# 数据抓取方案（API 启动后自动定时）

更新时间：2026-02-28

## 1. 目标

- API 子模块启动后自动开启定时抓取，不依赖 `systemd` 或外部 cron。
- 统一管理市场、天气、链上、信号等抓取任务，支持失败重试与降级。
- 暴露抓取监控指标，支持 Prometheus/Grafana 观测与告警。
- 保持与现有服务实现兼容（复用 `market/weather/chain/signal/player` service）。

## 2. 当前现状（基线）

当前项目是“手动触发 + CLI 循环”模式：

- 市场数据：`market sync`（支持 `--interval`）或 `POST /api/v1/markets/sync`
- 天气数据：`weather forecast/observe` 或 `GET /api/v1/weather/*` 按请求触发拉取并落库
- 链上扫描：`chain scan` 或 `chain watch`
- 信号生成：`signal generate` 或 `POST /api/v1/signals/generate`

`serve` 启动后未内置后台 scheduler。

## 3. 设计原则

- 复用优先：调度器只负责编排，不重写业务抓取逻辑。
- 幂等优先：任务可重复执行，依赖现有 upsert/去重规则。
- 可观测优先：每个 job 必须有运行次数、耗时、成功率、最新成功时间。
- 失效安全：单任务失败不影响其他任务；任务超时自动取消。
- 多实例可控：支持本地锁 + PostgreSQL advisory lock，防止重复执行。

## 4. 总体架构

新增 `internal/scheduler` 模块：

- `scheduler.Manager`
  - 注册任务定义（名称、间隔、超时、是否启动即执行、重试策略）
  - 管理 goroutine 生命周期（`Start(ctx)` / `Stop()`）
- `scheduler.JobRunner`
  - ticker + jitter 调度
  - 单任务并发保护（`TryLock`）
  - 执行超时控制（`context.WithTimeout`）
  - 运行结果上报到 metrics
- `scheduler.LockProvider`
  - `local`：单进程锁
  - `postgres_advisory`：跨实例锁（建议默认）

在 `serve` 启动流程中：

1. 初始化 DB 与 services
2. 初始化 scheduler manager（注入 service 依赖）
3. `go manager.Start(rootCtx)`
4. HTTP 服务退出时统一 cancel，等待 scheduler 优雅停止

## 5. 任务规划（MVP）

### 5.1 market_sync_job（必选）

- 间隔：5m
- 超时：90s
- 逻辑：调用 `market.Service.SyncMarkets(ctx)`
- 输出：市场表 upsert + `market_prices` 快照
- 失败重试：任务内 3 次指数退避（500ms 起步，最大 5s）

### 5.2 weather_forecast_job（必选）

- 间隔：15m
- 超时：120s
- 逻辑：
  - 读取活跃市场涉及城市（`markets` 去重 city）
  - 对每个城市调用 `weather.Service.GetForecast(... source=all, days=7)`
  - 城市级并发限制（建议 4~8）
- 输出：`forecasts` 表按 `(location, forecast_date, source)` upsert

### 5.3 chain_scan_job（必选）

- 间隔：1m~5m（按 RPC 配额调整）
- 超时：90s
- 逻辑：调用 `chain.Service.Scan(ctx, opts)`
- 输出：`competitors` / `competitor_trades`

### 5.4 signal_generate_job（建议）

- 间隔：10m
- 超时：60s
- 逻辑：调用 `signal.Service.GenerateSignals(ctx, limit)`
- 输出：`signals` 表（复用现有去重策略）

### 5.5 player_sync_job（建议）

- 间隔：30m
- 超时：60s
- 逻辑：调用 `player.Service.RefreshFromCompetitors(ctx)`
- 输出：`players` / `player_positions`

### 5.6 observation_job（可选）

- 间隔：10m
- 超时：60s
- 逻辑：对核心站点列表调用 `weather.Service.GetLatestObservation`
- 输出：`observations`

## 6. 配置草案（YAML）

```yaml
scheduler:
  enabled: true
  run_on_start: true
  lock_mode: postgres_advisory   # local | postgres_advisory
  lock_key_prefix: 7300
  default_timeout: 60s
  default_jitter_ratio: 0.1

  jobs:
    market_sync:
      enabled: true
      interval: 5m
      timeout: 90s
      immediate: true
    weather_forecast:
      enabled: true
      interval: 15m
      timeout: 120s
      immediate: true
      city_concurrency: 6
      days: 7
      source: all
    chain_scan:
      enabled: true
      interval: 2m
      timeout: 90s
      immediate: true
    signal_generate:
      enabled: true
      interval: 10m
      timeout: 60s
      immediate: false
      limit: 300
    player_sync:
      enabled: true
      interval: 30m
      timeout: 60s
      immediate: false

metrics:
  enabled: true
  path: /metrics
```

说明：

- 生产建议 `lock_mode=postgres_advisory`，避免多实例重复抓取。
- `immediate=true` 表示服务启动后立刻跑首轮，之后按 interval 执行。

## 7. 监控指标设计（Prometheus）

新增 `GET /metrics` 暴露以下指标。

### 7.1 调度层指标

- `sky_alpha_scheduler_job_runs_total{job,status}`
  - Counter；`status=success|error|timeout|skipped_locked`
- `sky_alpha_scheduler_job_duration_seconds{job,status}`
  - Histogram
- `sky_alpha_scheduler_job_inflight{job}`
  - Gauge；0/1
- `sky_alpha_scheduler_job_last_success_timestamp_seconds{job}`
  - Gauge
- `sky_alpha_scheduler_job_last_error_timestamp_seconds{job}`
  - Gauge
- `sky_alpha_scheduler_job_next_run_timestamp_seconds{job}`
  - Gauge

### 7.2 抓取层指标

- `sky_alpha_fetch_records_total{job,entity,result}`
  - Counter；例如 `entity=markets|prices|forecasts|observations|trades|signals`
- `sky_alpha_external_request_total{provider,operation,status_code}`
  - Counter；provider 如 `gamma|clob|nws|openmeteo|visualcrossing|polygon_rpc|vertex_ai`
- `sky_alpha_external_request_duration_seconds{provider,operation}`
  - Histogram
- `sky_alpha_data_freshness_seconds{dataset}`
  - Gauge；例如 `dataset=markets|forecasts|chain|signals`

## 8. 告警基线（建议）

- `market_sync_job` 连续 3 次失败告警
- 任意 job `last_success` 超过 `3 x interval` 告警
- `job_duration p95 > 0.8 * timeout` 告警
- `data_freshness_seconds` 超阈值告警：
  - markets > 20m
  - forecasts > 1h
  - chain > 15m
  - signals > 30m

## 9. 并发与一致性策略

- 单任务防重入：同一进程内 `TryLock`
- 跨实例防重入：PostgreSQL advisory lock（按 job 名称映射 lock key）
- 超时取消：每次执行都绑定 timeout context
- 失败隔离：A 任务失败不阻塞 B 任务
- 幂等写入：继续依赖现有 upsert/unique 约束

## 11. 验收标准

- API 启动后 1 分钟内可见至少一个 job 执行记录
- `/metrics` 能看到每个 job 的 runs/duration/last_success
- 关闭服务后 scheduler goroutine 在 10 秒内退出
- 多实例部署时，同一 job 同一时刻只允许一个实例执行
- 连续故障时有明确告警信号，不出现静默停摆
