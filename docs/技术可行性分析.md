# Sky Alpha Pro 技术可行性分析

> 评估视角：技术架构师  
> 评估对象：`docs/产品整体架构.md`（v0.1.0, 2026-02-28）  
> 结论日期：2026-02-28

---

## 1. 执行摘要

基于当前架构方案，**项目整体技术上可行**，且能够在 MVP 范围内落地。  
但按现有 12 周排期，若团队规模较小（<4 名全职工程师）或缺少 Polymarket/链上交易经验，存在明显延期风险。

**结论：有条件可行（Conditional Go）**

成立条件：
1. MVP 严格聚焦温度市场，不扩展非温度品类。
2. Phase 5（链上分析+玩家追踪）按“基础版”交付，深度策略分析后置。
3. 提前完成 EIP-712 签名、CLOB 沙盒联调与交易风控回测。
4. 引入统一可观测性（日志、指标、告警）作为上线前硬门槛。

---

## 2. 评估范围与方法

评估覆盖以下维度：
- 架构分层与模块边界合理性
- 外部依赖可达性与集成复杂度
- 数据模型支撑能力
- 非功能需求（性能、稳定性、安全、可运维性）
- 里程碑排期可达性
- 关键技术风险与缓释方案

评估依据：
- 文档中的技术栈、模块设计、数据表结构、API/CLI、里程碑与外部依赖说明。

## 3. 架构可行性评估

### 3.1 分层架构

分层（接入层/Agent 层/服务层/执行层/数据层/集成层）职责清晰，具备良好的可演进性。  
Go + Gin + GORM + PostgreSQL 的组合适合 MVP 快速交付。

可行性判断：**高**

### 3.2 AI Agent（Gemini + Function Calling）

技术路径可行，8 个 Tool 的定义粒度适中，覆盖“数据获取-估值-决策-执行”闭环。  
核心风险不在“能否接入模型”，而在“可控性”：
- Tool 调用链稳定性与超时处理
- 推理输出结构化约束（防止非预期交易指令）
- 成本与时延波动

建议：
- 对 `place_order` 设置二次校验（服务端硬规则优先于模型建议）。
- 推理输出采用强 schema（JSON schema + strict parser）。
- 引入“回放模式”验证 Agent 决策一致性。

可行性判断：**中高**

### 3.3 WeatherService（NWS/Open-Meteo/Visual Crossing）

三级数据源设计合理，兼顾解析源权威性与预报多样性。  
主要挑战是多源时间对齐、单位一致性、观测口径差异。

建议：
- 统一内部时间基准（UTC 存储 + 市场时区转换）。
- 统一单位转换层（°F/inch/mph），避免业务逻辑层重复换算。
- 对每个市场固化“解析站点映射”，避免动态匹配漂移。

可行性判断：**高**

### 3.4 MarketService（Gamma/CLOB/WebSocket）

市场发现与交易执行分层合理。  
复杂点主要在 CLOB 认证签名与订单状态一致性（REST + WS）。

建议：
- 建立“订单状态机”并以用户 WS 回报为准做最终一致。
- 对签名、nonce、timestamp 做统一封装，避免多处实现。
- 先完成只读链路（行情+订单簿）再接入实盘下单。

可行性判断：**中高**

### 3.5 ChainAnalyzer 与 PlayerTracker

功能价值高，但对 MVP 的“首要盈利闭环”并非必需。  
链上事件监听、地址聚类、机器人识别属于高不确定性模块，研发与验证成本高。

建议：
- MVP 中仅交付基础扫描与统计，不做复杂图分析与策略归因。
- 将“机器人检测准确率”作为实验性指标，不作为交易硬依赖。

可行性判断：**中（基础版可行，完整版偏高风险）**

### 3.6 TradingEngine 与风控

风控参数定义完备，且 `confirmation_required=true` 符合 MVP 安全策略。  
关键在于“风控必须在 Agent 之外强制执行”。

建议：
- 服务端硬性校验：仓位、日亏、流动性、最小 edge、重复下单抑制。
- 引入 kill switch（全局停机开关）与按市场熔断。
- 交易前后持仓快照 + 审计日志不可变记录。

可行性判断：**高**

## 4. 数据模型可行性评估

现有表设计覆盖市场、价格、预报、观测、信号、交易、竞争对手、玩家、Agent 日志，完整度高。  
对 MVP 而言，存在“过度前置”的迹象，但不影响可行性。

关键优化建议：
1. 对高频表 `market_prices` 与 `observations` 设计分区或保留策略（如 90 天热数据）。
2. `signals.reasoning`、`agent_logs.reasoning` 建议脱敏与长度限制，防止成本与隐私风险。
3. 为 `trades(order_id)`、`competitor_trades(tx_hash)` 增加唯一约束，避免重复写入。
4. 明确金额字段精度规范（USDC 6 位小数）并在应用层统一量化。

可行性判断：**高**

---

## 5. 非功能可行性评估

### 5.1 性能与容量

MVP 规模下（城市数有限、市场数有限）Go + PostgreSQL 足够。  
瓶颈主要来自外部 API 延迟与限频，而非内部计算。

建议指标基线：
- 行情采集成功率 ≥ 99%
- Agent 分析 P95 时延 < 8s
- 下单链路成功率 ≥ 99.5%（非风控拒绝场景）

### 5.2 稳定性与容错

文档已包含重试方向，但需制度化：
- 外部 API 客户端统一超时/重试/熔断策略
- 任务调度幂等化（防止重复抓取、重复下单）
- WS 断线自动重连 + 增量补偿

### 5.3 安全与合规

涉及私钥、交易、第三方 API，安全要求高：
- 私钥必须托管于 KMS/HSM 或最少采用密文+内存最短驻留
- 配置脱敏与审计日志分级
- API Key 与交易操作权限分离

### 5.4 可观测性

当前文档仅描述日志。建议补齐：
- Metrics: QPS、错误率、外部 API 延迟、订单成功率、PnL 波动
- Tracing: Agent tool-call 链路追踪
- Alerting: 日亏阈值、下单失败率、数据采集中断

综合判断：**非功能层面可行，但需在 Phase 6 前置部分关键能力**

---

## 6. 里程碑可达性评估（12 周）

原计划逻辑完整，但对“并行开发能力”依赖高。  
主要风险点：Phase 3（Agent）与 Phase 4（实盘交易）耦合过紧，Phase 5 复杂度偏高。

建议调整：
1. 将 `EIP-712 + CLOB 实单演练` 提前到 Phase 3 末期完成联调门槛。
2. Phase 5 拆分为“基础监控（MVP内）”与“高级分析（MVP后）”。
3. 在 Phase 2 增加“数据质量验收周”，确保后续 Agent 输入稳定。

排期结论：
- 4-6 人工程团队：12 周可达（中等风险）。
- 2-3 人工程团队：建议 14-16 周（高风险）。

## 7. 关键技术风险与缓释

| 风险项 | 概率 | 影响 | 缓释策略 |
|---|---|---|---|
| CLOB 签名/下单联调不稳定 | 中高 | 高 | 建立独立交易沙盒测试集，先只读后下单 |
| 多源天气数据口径不一致 | 高 | 中高 | 统一单位/时区/站点映射，建立数据质量规则 |
| Agent 输出不可控导致误触发 | 中 | 高 | 服务端硬风控 + JSON schema 强约束 + 人工确认 |
| 外部 API 限频或抖动 | 中 | 中高 | 缓存、退避重试、分级降级策略 |
| 链上分析模块超期 | 高 | 中 | 降级为基础扫描，复杂识别后置 |
| 私钥与凭证安全风险 | 中 | 高 | 密钥托管、最小权限、审计与轮换机制 |

---

## 8. 架构师建议的 MVP 最小闭环

建议将 MVP 的“必须完成”聚焦为：
1. 市场同步 + 天气多源采集 + 数据入库
2. 概率估计 + Edge 计算 + 信号输出
3. 风控校验 + 人工确认下单 + 持仓/PnL
4. 基础监控与告警（至少覆盖交易和采集链路）

建议延后到 MVP+1：
1. 高级机器人识别（地址图谱、策略归因）
2. 深度玩家策略模仿与自动进化
3. 全自动无人值守交易

---

## 9. Go / No-Go 判定

当前建议：**Go（附条件）**

上线前必须满足的硬门槛：
1. 连续 2 周小额实盘无风控失效事件。
2. 数据采集与下单主链路错误率低于阈值（可观测可追踪）。
3. 出现异常时可在 1 分钟内触发全局停机（kill switch）。
4. 关键密钥管理与审计机制通过内部安全检查。

在满足上述门槛前，不建议切换到无人值守自动交易模式。
