# 第一阶段改造清单（聚焦：定时唤醒 + 记忆能力）

> 目标：先把“Agent 主控”跑稳，再逐步扩展策略复杂度。  
> 原则：先控制平面，后策略平面。

---

## 1. 阶段目标

本阶段只解决四件事：

1. 定时任务稳定唤醒 Vertex AI Agent
2. 每轮只调用一次 LLM 主接口（`gemini-2.5-pro`）
3. 建立可用的 MVP 记忆闭环（跨轮复盘）
4. 打通 Polymarket 市场文本中的城市解析（question/slug -> city）

---

## 2. 里程碑

### M1：Scheduler -> Agent 单轮唤醒链路

交付项：

- 新增 Agent 调度任务（如 `agent_cycle`）
- 防重入锁与超时控制
- 单轮预算配置（runtime/tool calls/external requests）
- 统一运行状态写入 `scheduler_runs`
- 城市解析工具纳入标准工具目录（批量模式）

验收标准：

- 定时任务可稳定触发
- 同时只运行一个 agent cycle
- 调度失败可重试且可观测
- 单轮计划中必须显式包含“城市解析”步骤（当存在 unresolved markets）

---

### M2：单轮单次 LLM 调用

交付项：

- Go 侧“上下文聚合器”
- 单次 `generateContent` 调用获取结构化计划
- 计划执行器（按计划调用 MCP tools）
- 本轮 LLM 调用计数指标
- `market.city.resolve.batch` 工具（输入 markets，输出 city + confidence + source）

验收标准：

- 每轮 `llm_calls == 1`
- Tool 调用由计划驱动，不再硬编码串联
- 失败可定位（plan 解析失败/工具执行失败）
- `markets_spec_ready` 在有可解析市场时不再长期为 0

---

### M3：MVP 记忆闭环

交付项：

- `agent_memories`（轮次摘要）
- 记忆写入器：轮次结束时生成结构化摘要
- 记忆读取器：下一轮注入最近 N 轮摘要
- 记忆字段：
  - key_failures
  - funnel_summary
  - action_suggestions
  - execution_outcome

验收标准：

- 下一轮 Prompt 可读取前轮摘要
- 同类错误复发率可观测
- 能看到“基于上轮复盘”的计划调整

---

## 3. 第一阶段不做的事情

- 不做复杂外部记忆系统接入（向量数据库/图记忆）
- 不做策略模型重训练
- 不做前端可视化系统

说明：先把最小记忆闭环跑通，再扩展外部记忆架构。

---

## 3.1 Polymarket 城市解析专项（必须落地）

问题背景：

- 目前信号链路的第一断点是 `spec_ready=0`
- 根因通常是 `market.city` 为空，且 question/slug 未被稳定解析

本阶段要求：

1. 新增/统一城市解析工具：`market.city.resolve.batch`
2. 解析输入必须包含：`market_id/question/slug/market_type`
3. 解析输出必须包含：`city/source/confidence/skip_reason`
4. 解析结果必须写入缓存（正缓存 + 负缓存）
5. 解析失败必须结构化记录（不可只留日志）

验收 SQL（核心）：

- `markets_spec_ready` 比例连续上升
- `signal_runs.spec_ready` 不再连续为 0
- `top_skip_reasons` 中 `city_missing` 占比下降

---

## 4. 记忆架构扩展预留

### 4.1 预留接口

- `MemoryStore` 接口抽象：
  - `Write(sessionSummary)`
  - `Query(context)`
  - `Compact(policy)`

### 4.2 未来可接入方向（调研结论）

- SQL + 向量混合检索
- 分层记忆（短期/长期/语义）
- 多 Agent 共享记忆总线

### 4.3 当前建议

- 阶段 1 用 PostgreSQL 存结构化摘要
- 保持接口稳定，后续替换底层实现

---

## 5. 关键配置项（阶段 1）

- `agent_cycle.enabled`
- `agent_cycle.interval`
- `agent_cycle.timeout`
- `agent_cycle.max_tool_calls`
- `agent_cycle.max_external_requests`
- `agent_cycle.memory_window`（最近 N 轮）

---

## 6. 验收指标

### 技术稳定性

- `agent_cycle_success_rate > 95%`
- `llm_calls_per_cycle = 1`
- `tool_error_rate` 持续可控

### 业务有效性

- 漏斗指标可持续产出：
  - `markets_total/spec_ready/forecast_ready/raw_edge_pass/exec_edge_pass/signals_generated`
- 无输入时有明确 skip reason，不静默失败

### 复盘有效性

- 记忆注入后，同类错误复发率下降
- 计划中出现“针对上轮失败点”的动作

---

## 7. 风险与回滚

### 风险

- Prompt 输出结构不稳定
- Vertex 短时波动影响单轮任务
- 记忆摘要质量不足导致误导

### 回滚

- 一键切回 rule-only 主流程
- 关闭 `agent_cycle`，保留底层工具可手动触发
- 保留所有审计数据用于复盘

---

## 8. 完成定义（DoD）

满足以下条件视为第一阶段完成：

1. 定时任务可稳定唤醒 Agent。  
2. 单轮任务只调用一次 LLM。  
3. 有可用记忆闭环（写入+读取+注入）。  
4. 运维可通过日志与指标定位失败原因。  
5. 保留回滚路径且可验证。
