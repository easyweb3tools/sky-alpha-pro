# Sky Alpha Pro 产品功能模块划分及开发计划

> 文档版本：v0.1  
> 适用阶段：MVP（12 周）  
> 基线日期：2026-02-28

---

## 1. 目标与范围

本计划用于将 Sky Alpha Pro 的 MVP 需求拆解为可执行的功能模块和开发任务，确保在 12 周内形成可实盘小额运行的系统闭环。

MVP 目标闭环：
1. 获取天气市场与气象数据。
2. 生成可解释交易信号。
3. 在风控约束下执行交易（人工确认）。
4. 持续追踪持仓、订单、PnL 与系统健康状态。

## 2. 功能模块划分

|---|---|---|---|

| M02 | 市场数据服务（Polymarket） | 核心业务 | P0 |

| M05 | AI Agent 编排层 | 核心业务 | P0 |
| M06 | 交易执行与风控引擎 | 核心业务 | P0 |
| M07 | 持仓/PnL 与审计日志 | 核心业务 | P0 |
| M08 | API 与 CLI 接入层 | 交互层 | P0 |
| M09 | 调度与任务编排 | 运行层 | P0 |

| M12 | 可观测性与运维保障 | 保障能力 | P0 |

### 2.2 模块边界与职责

#### M01 平台基础与工程框架（P0）
- 职责：项目骨架、配置、日志、数据库连接、迁移、环境管理。
- 输入：配置文件、环境变量。
- 输出：统一工程运行底座。
- 依赖：PostgreSQL、Viper、Zap、GORM。

#### M02 市场数据服务（P0）
- 职责：对接 Gamma/CLOB，拉取市场、价格、订单簿、实时行情。
- 输入：市场筛选条件、token/market id。
- 输出：标准化市场实体与价格快照。
- 依赖：Polymarket Gamma API、CLOB API、WS。

#### M03 气象数据服务（P0）
- 职责：对接 NWS/Open-Meteo/Visual Crossing，汇总预报与观测。
- 输入：城市/经纬度/站点、时间窗口。
- 输出：标准化 forecast/observation 数据。
- 依赖：weather.gov、open-meteo、visual crossing。

#### M04 概率估计与信号引擎（P0）
- 职责：多源预报融合、概率估计、edge 计算、信号入库。
- 输入：市场价格、天气数据、解析规则。
- 输出：信号（方向、edge、置信度、理由）。
- 依赖：M02、M03、规则配置。

#### M05 AI Agent 编排层（P0）
- 职责：封装 tool calling，编排分析流程，输出分析结论。
- 输入：分析请求（全量或单市场）。
- 输出：结构化分析结果与 tool 调用链。
- 依赖：Gemini SDK/ADK、M02/M03/M04/M06。

#### M06 交易执行与风控引擎（P0）
- 职责：订单构建、EIP-712 签名、下单/撤单、风控硬校验。
- 输入：信号与交易参数。
- 输出：订单状态、执行结果、异常告警。
- 依赖：CLOB 认证、M07。

#### M07 持仓/PnL 与审计日志（P0）
- 职责：订单与成交落库、持仓快照、PnL 统计、审计追踪。
- 输入：订单状态流、市场结果。
- 输出：可查询的持仓与盈亏报告。
- 依赖：M02、M06。

#### M08 API 与 CLI 接入层（P0）
- 职责：提供 REST 与 CLI 统一入口。
- 输入：外部调用请求。
- 输出：结构化响应、命令执行反馈。
- 依赖：Gin、cobra、全部业务模块。

#### M09 调度与任务编排（P0）
- 职责：定时同步市场/天气/扫描任务；失败重试与幂等控制。
- 输入：cron 配置。
- 输出：周期任务执行记录。
- 依赖：robfig/cron、M02/M03/M10/M11。

#### M10 链上扫描与竞争对手监控（P1 基础版）
- 职责：监听核心合约事件，识别高频地址，输出可疑活动列表。
- 输入：区块/事件流。
- 输出：competitor 与 competitor_trades 数据。
- 依赖：Polygon RPC、go-ethereum。

#### M11 玩家追踪（P1 基础版）
- 职责：抓取玩家榜单与持仓，提供基础比较能力。
- 输入：Data API/链上活动。
- 输出：players 与 player_positions。
- 依赖：Polymarket Data API、M10（可选）。

#### M12 可观测性与运维保障（P0）
- 职责：指标、告警、链路日志、故障降级、kill switch。
- 输入：系统运行事件。
- 输出：运行健康视图与告警信号。
- 依赖：日志系统、监控系统。

## 3. 开发里程碑与阶段计划

## 3.1 阶段划分（12周）

| 阶段 | 周期 | 核心目标 | 对应模块 |

| Phase A | Week 1-2 | 工程底座可运行 | M01, M08(骨架), M12(基础) |
| Phase B | Week 3-4 | 数据采集闭环 | M02, M03, M09 |
| Phase C | Week 5-6 | 分析与信号闭环 | M04, M05 |

| 周次 | 关键任务 | 产出物 |

## 4. 关键依赖与并行策略

### 4.1 关键依赖关系

1. M01 完成后，M02/M03/M08/M09 才能稳定并行。
2. M04 依赖 M02+M03 数据质量，需在 W4 后进入主开发。
3. M06 依赖 M02（CLOB）与 M07（账务落库）联调。
4. M10/M11 不应阻塞交易主链路上线。

### 4.2 并行开发建议（4-6人团队）

| 角色 | 主责模块 | 次责模块 |
|---|---|---|
| Tech Lead/架构 | M01, M12, 跨模块设计评审 | M06 风控规则 |
| 后端A（行情） | M02, M09 | M08 市场接口 |
| 后端B（天气） | M03, M04 | M08 天气接口 |
| 后端C（交易） | M06, M07 | M02 CLOB 联调 |
| AI工程师 | M05, M04 推理与提示词 | M12 Agent 观测 |
| 数据/链上工程师 | M10, M11 | M07 报表辅助 |

## 5. 验收标准（Definition of Done）

### 5.1 模块级验收

| 模块 | 最低验收标准 |
|---|---|
| M02 | 市场同步成功率 >= 99%，价格快照完整入库 |
| M03 | 三数据源可切换，单位/时区一致，异常可降级 |
| M04 | 同一输入结果可复现，信号包含 edge+confidence+reasoning |
| M05 | Tool 调用链可追踪，输出满足结构化 schema |
| M06 | 下单前强风控校验，支持人工确认，支持撤单 |
| M07 | 持仓与PnL可按日查询，订单状态最终一致 |
| M12 | 核心错误告警、kill switch、关键日志字段齐全 |

### 5.2 MVP 上线前验收

1. 连续 7 天稳定运行，无 P0 级故障。
2. 小额实盘回放与真实交易结果一致性通过。
3. 主链路（采集→信号→下单→记账）可全程追踪。
4. 密钥管理、配置脱敏、审计日志通过内部检查。

## 6. 风险清单与应对计划

| 风险 | 触发信号 | 应对措施 | 负责人 |
|---|---|---|---|
| CLOB 下单联调延期 | W6 仍无法稳定下单 | 提前建立 mock + 沙盒演练，W5 开始预联调 | 交易负责人 |
| 数据质量不稳定 | 信号波动异常、缺失率升高 | 增加数据质量任务与阈值告警，阻断坏数据入模 | 行情/天气负责人 |
| Agent 成本或时延超预期 | P95 > 目标，成本快速上升 | 增加缓存、模板化推理、分层模型调用 | AI负责人 |
| 链上模块超期 | W10 未达到基础能力 | 降级为离线批处理，不阻塞 MVP 主线 | 链上负责人 |
| 安全风险 | 密钥泄露或误操作 | 密钥托管、最小权限、双重确认、kill switch | Tech Lead |

## 7. 发布策略

### 7.1 环境策略

1. `dev`：本地联调与功能验证。  
2. `staging`：接近生产的数据流与策略演练。  
3. `prod`：小额实盘，严格风控与人工确认。

### 7.2 发布节奏

1. 每周一次小版本发布（固定窗口）。  
2. 交易相关变更必须经过 staging 回放测试。  
3. 生产启用灰度开关（城市、市场类型、最大仓位逐步放开）。

## 9. 结论

该模块划分支持“先交易主链路、后增强能力”的交付策略，能够在控制风险的前提下实现 MVP 可上线目标。  
建议严格执行 P0/P1 边界与阶段验收门槛，避免在 Week 7 后引入范围膨胀导致延期。
