# 可观测性与运维可诊断性实现清单

## 1. 背景与目标

当前系统存在“服务健康但业务停摆”的风险：

- HTTP `/health` 正常，但抓取链路可能无数据增长。
- 失败原因主要在日志文本中，缺少结构化沉淀，运维定位成本高。
- 配置生效状态不透明（例如环境变量未注入、任务未注册、参数回落默认值）。

本清单目标：

1. 让运维在 5 分钟内判断“哪个任务失败、为什么失败、从什么时候开始失败”。
2. 让开发在不翻日志的前提下，通过 SQL/API/Prometheus 完成排障闭环。
3. 将“无输入/被限流/上游失败/配置缺失”等状态显式化，避免静默停摆。

---

## 2. 实施范围

- 调度层：`internal/scheduler/*`
- 抓取任务：`internal/scheduler/jobs.go`
- 业务服务：`internal/market`, `internal/weather`, `internal/chain`
- 指标：`pkg/metrics/registry.go`
- 运维接口：`internal/server/*`
- 模型与迁移：`internal/model/models.go`, `pkg/database/auto_migrate.go`
- 文档：`README.md`, `docs/数据抓取方案.md`

## 3. P0（必须先做）

### 3.1 新增调度执行台账表 `scheduler_runs`

- [ ] 在 `internal/model/models.go` 增加模型：`SchedulerRun`
- [ ] 字段建议：
  - `id`
  - `job_name`
  - `started_at`, `finished_at`, `duration_ms`
  - `status`（`success|error|timeout|skipped_locked|skipped_no_input`）
  - `records_success`, `records_error`, `records_skipped`
  - `error_code`, `error_message`
  - `meta_json`（jsonb）
  - `created_at`
- [ ] 纳入 `AutoMigrate`

验收：

- 每次 job 执行后落 1 条记录（成功与失败都要落）。
- 可用 SQL 直接查看最近失败：

```sql
SELECT job_name, status, error_code, error_message, started_at
FROM scheduler_runs
ORDER BY id DESC
LIMIT 50;
```

### 3.2 统一 Job 结果结构（从文本错误升级为结构化）

- [ ] 扩展 `scheduler.JobResult`：
  - `Records []FetchRecord`
  - `Freshness map[string]float64`
  - `Errors []JobError`
  - `Warnings []JobWarning`
  - `SkipReason string`
- [ ] 新增错误码枚举（示例）：
  - `empty_city_set`
  - `chain_rpc_empty`
  - `rpc_range_limit`
  - `upstream_429`
  - `upstream_5xx`
  - `clob_midpoint_failed`
  - `db_write_failed`

验收：

- 关键失败可按 `error_code` 聚合，不依赖 grep 日志。

### 3.3 增加核心 Prometheus 指标

- [ ] `sky_alpha_scheduler_job_runs_total{job,status}`
- [ ] `sky_alpha_scheduler_job_duration_seconds{job,status}`
- [ ] `sky_alpha_scheduler_job_consecutive_failures{job}`
- [ ] `sky_alpha_scheduler_job_last_success_timestamp_seconds{job}`
- [ ] `sky_alpha_scheduler_job_last_error_timestamp_seconds{job}`
- [ ] `sky_alpha_scheduler_records_total{job,entity,result}`
- [ ] `sky_alpha_scheduler_errors_total{job,error_code}`
- [ ] `sky_alpha_data_freshness_seconds{dataset}`（接线到真实数据）

验收：

- `/metrics` 中可看到上述指标。
- 连续失败次数可随失败累加，成功后归零。

### 3.4 增加运维状态接口 `GET /ops/status`

- [ ] 新建内部运维 handler，输出：
  - scheduler 总状态（enabled、start_time）
  - 每个 job：`enabled/interval/next_run/inflight/last_status/last_success/last_error/consecutive_failures/last_error_code`
  - 数据集 freshness：`markets/prices/forecasts/chain/signals`
  - 关键配置生效值（脱敏）：
    - `chain.rpc_url_configured`（bool）
    - `agent.vertex_project_configured`（bool）
    - `weather.visualcrossing_key_configured`（bool）
    - `chain.scan_lookback_blocks`（number）

验收：

- 运维无需进容器即可看见“配置是否生效 + 任务是否在跑 + 最近错误原因”。

### 3.5 显式化“无输入”状态，禁止静默成功

- [ ] `weather_forecast`：当 `loadActiveCities` 返回空，记录 `skipped_no_input`，`error_code=empty_city_set`
- [ ] `chain_scan`：当 `rpc_url` 为空，记录 `skipped_no_input`，`error_code=chain_rpc_empty`
- [ ] `market_sync`：当抓到 0 条市场时，记录 `records_skipped`

验收：

- 不再出现“任务看起来成功，实际啥也没做”的假象。

## 4. P1（建议尽快做）

### 4.1 启动期配置生效快照

- [ ] 应用启动时输出一条结构化日志：
  - 已注册 job 列表
  - 每个 job interval/timeout/immediate
  - 关键配置是否已设置（脱敏）
- [ ] 可选：将配置快照写入 `scheduler_runs.meta_json` 的首条记录

验收：

- 可用单条日志快速确认“env 是否生效，任务是否注册”。

### 4.2 健康检查分层

- [ ] 保留 `/health`（进程存活）
- [ ] 新增 `/ready`（业务就绪）
  - 市场任务最近 30 分钟内至少成功一次
  - 天气任务最近 60 分钟内成功且有有效抓取
  - 若配置了 RPC，则链扫描最近 10 分钟内成功

验收：

- 监控系统可直接基于 `/ready` 告警“活着但停摆”。

### 4.3 提供运维 SQL 视图（可选）

- [ ] 新建视图 `ops_job_latest_status`（每个 job 最近一次执行状态）
- [ ] 新建视图 `ops_data_freshness`（各数据集最新时间与延迟）

- [ ] 手动制造 `chain rpc empty`，`/ops/status` 能显示错误码。
- [ ] 手动制造 `weather city empty`，状态为 `skipped_no_input`，非 success。
- [ ] 正常运行 30 分钟后，`scheduler_runs` 有完整成功/失败记录。

- [ ] 运维可仅用以下三步定位问题：
  1. `GET /ops/status`
  2. `SELECT ... FROM scheduler_runs ORDER BY id DESC LIMIT 50;`

## 7. 实施顺序建议（两次迭代）

### 迭代 A（1-2 天）

- [ ] `scheduler_runs` 表 + manager 记录执行结果
- [ ] JobResult 结构化错误
- [ ] `/ops/status` 最小版本

### 迭代 B（1-2 天）

- [ ] 指标补齐 + consecutive failure
- [ ] `/ready` 业务就绪检查
- [ ] 文档更新（README + 数据抓取方案）

---

## 8. 说明

- MVP 阶段优先“可诊断性”而非“全自动修复”。
- 所有状态都应可查询、可观测、可告警，不依赖人工翻日志。
- 本清单落地后，出现“线上不报错但抓不到数据”时，应能在 5 分钟内定位到具体任务和错误码。
