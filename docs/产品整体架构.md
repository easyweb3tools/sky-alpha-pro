# Sky Alpha Pro 产品整体架构（Agent-First / MCP-First）

> 版本：v2.1（重构版）  
> 日期：2026-03-03

---

## 1. 架构定位

Sky Alpha Pro 从“Go 主流程 + AI 辅助”升级为“AI 主控 + Go 执行”：

- `Vertex AI Agent`：主控大脑，负责任务规划与流程编排
- `Go API / CLI / Service`：MCP Tool Provider，被动执行工具能力
- `Go 风控与审计`：强约束层，保障可控与可追溯

一句话：

> Agent 决策与编排，Go 执行与约束。

---

## 2. 核心原则

1. Agent 是控制平面，Go 是数据平面。  
2. 高频能力必须批量化（batch tool），禁止单条循环调用。  
3. 单轮任务只调用一次 LLM 主决策接口。  
4. 关键写操作（交易/状态变更）必须经过 Go 风控。  
5. 全链路可观测、可审计、可回放。

---

## 3. 总体架构

```text
┌────────────────────────────────────────────────────────────┐
│                 Scheduler Wakeup Layer (Go)               │
│  cron/interval trigger + lock + budget + lifecycle        │
└────────────────────────────────────────────────────────────┘
                          │ trigger
                          ▼
┌────────────────────────────────────────────────────────────┐
│                Vertex AI Agent (gemini-2.5-pro)           │
│  - receives cycle context + memory summary                │
│  - one LLM call per cycle                                 │
│  - outputs plan + tool sequence + guardrails              │
└────────────────────────────────────────────────────────────┘
                          │ tool-calling (MCP)
                          ▼
┌────────────────────────────────────────────────────────────┐
│                 Go MCP Tool Runtime (Passive)             │
│ market/weather/signal/trade/chain/player/report tools     │
└────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌────────────────────────────────────────────────────────────┐
│              Memory + Audit + State (PostgreSQL)          │
│ agent_sessions/tool_calls/agent_memories/signals/trades   │
└────────────────────────────────────────────────────────────┘
```

## 4. 定时任务与 Agent 唤醒机制（新增）

### 4.1 调度职责

由 Go Scheduler 负责：

- 定时唤醒（如每 5/15 分钟）
- 单实例防重入（分布式锁或进程锁）
- 周期预算（最大运行时长、最大工具调用数）
- 失败重试与退避（不直接重试 LLM 多次）

### 4.2 单轮执行约束

每轮（一次 scheduler trigger）严格执行：

1. 组装本轮上下文快照（市场、天气、链路、上轮摘要）
2. 调用 Vertex AI **一次**（`gemini-2.5-pro`）
3. 获取结构化计划（Plan）
4. 按计划调用 MCP tools（可多次）
5. 写入会话结果与记忆摘要

### 4.3 为什么这样设计

- 避免“每个城市一次 LLM 调用”的碎调用模式
- 将 LLM 调用变成低频高价值决策调用
- 显著降低 429 风险与调用成本波动

## 5. Agent 记忆层（新增）

> 目标：让 Agent 根据上次执行结果复盘并优化下一轮安排。

### 5.1 MVP 记忆分层

1. `Session Memory`（短期）
- 当前会话中的步骤状态、错误、关键决策

2. `Episodic Memory`（中期）
- 轮次级总结：成功率、失败原因、漏斗变化、建议动作
- 存在 PostgreSQL，供下一轮 Prompt 注入

3. `External Memory Adapter`（扩展）
- 预留接口，后续可接入成熟外部记忆系统（向量/图谱/混合记忆）

### 5.2 记忆写入与读取

- 写入时机：每轮任务结束（成功/失败都写）
- 读取时机：下一轮任务开始，注入“最近 N 轮摘要”
- 注入内容：
  - 上轮失败原因 TopN
  - 上轮漏斗短板（spec_ready/forecast_ready/exec_edge_pass）
  - 上轮建议执行动作与实际效果

### 5.3 记忆质量原则

- 记忆是“结构化事实摘要”，不是原始长文本堆积
- 必须可追溯到具体 session 与 tool call
- 记忆过期与压缩策略可配置

## 6. MCP 工具层职责

Go 工具层只做能力提供，不做复杂编排：

- `market.sync.batch`
- `market.cities.active`
- `weather.forecast.batch`
- `market.price.latest.batch`
- `signal.generate.batch`
- `trade.simulate.batch` / `trade.place.batch`
- `chain.scan.batch`

统一响应结构：

- `status`: ok/degraded/error
- `data`
- `metrics`
- `errors[]`（含 `error_code`）

## 7. Prompt 工程

### 7.1 系统提示职责

- 明确 Agent 是调度器
- 必须优先 batch 工具
- 必须引用记忆摘要进行计划调整
- 必须输出结构化计划（可被执行器解析）

### 7.2 运行护栏

- 每轮最多 1 次 LLM 主调用
- 每轮最大 Tool 调用次数
- 每轮最大外部请求预算
- 写操作必须经过风控工具

## 9. 可观测性与验收指标

### 9.1 Agent 维度

- session 成功率
- step 成功率
- tool 错误码分布
- 单轮 LLM 调用次数（应为 1）

### 9.2 业务维度

- `markets_total/spec_ready/forecast_ready/raw_edge_pass/exec_edge_pass/signals_generated`
- 信号到交易转化率
- PnL 与回撤

### 9.3 记忆维度

- 记忆命中率（是否被下一轮引用）
- 同类错误复发率（是否下降）
- 计划调整有效性（是否改善漏斗）

---

## 10. 风险与降级

- Vertex 不可用或限流：
  - 进入 `rule-only` 或 `cached-plan` 降级
  - 本轮不中断，记录 degraded 状态
- Prompt 漂移：
  - Prompt 版本化 + 灰度
- 执行风险：
  - 所有交易仍由 Go 风控硬约束

---

## 11. 迁移策略

双轨迁移：

- Track A：现有规则流程（兜底）
- Track B：Agent-First 流程（灰度）

切换门槛：

- 连续 7 天 session 成功率 > 95%
- 单轮 LLM 调用稳定为 1
- 信号产出与执行质量不低于 Track A

---

## 12. 结论

本架构的关键不是“用不用 AI”，而是“AI 处在控制层还是接口层”。

最终形态：

- Vertex AI 是大脑（主控）
- Go 是手脚（MCP 工具）
- 记忆是反馈回路（持续复盘优化）
