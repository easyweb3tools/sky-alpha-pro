# Sky Alpha Pro 产品整体架构（目标态 / Agent-First）

> 版本：v2.3（目标态）  
> 日期：2026-03-05

---

## 1. 架构定位

Sky Alpha Pro 的目标是“AI 主控决策系统”，不是“AI 增强脚本系统”：

- `Vertex AI Agent`：控制平面（决策、编排、复盘、策略迭代）
- `Go API / CLI / Service`：执行平面（MCP 工具能力、风控、审计）
- `PostgreSQL + Metrics`：事实平面（状态、结果、证据、指标）

一句话：

> Vertex 决策，Go 执行，数据闭环驱动进化。

---

## 2. 核心原则（目标架构约束）

1. Agent 是控制平面，Go 是数据/执行平面。  
2. 工具调用优先 batch，禁止 N×单条外部调用。  
3. 预算约束优先于“调用次数约束”：按 `token/tool/time/external` 预算治理单轮运行。  
4. 交易与状态变更必须经过 Go 风控硬约束。  
5. 验收拆分为两条线：`系统健康验收` 与 `策略价值验收`。  
6. 故障必须分层归因：配置层、数据层、策略层、执行层。

---

## 3. 总体架构（目标态）

```text
┌────────────────────────────────────────────────────────────┐

```

## 4. 关键架构改进（非代码细节）

### 4.1 从“单次 AI 调用”升级为“单轮会话预算”

目标不是机械限制“一次 HTTP 调用”，而是限制单轮资源消耗：

- `max_tokens_per_cycle`
- `max_tool_calls`
- `max_external_requests`
- `max_cycle_duration`

收益：

- 在复杂轮次下保持可控而不僵化
- 避免“调用次数满足但预算失控”的伪合规

### 4.2 目标函数前置（价值导向）

每轮必须绑定可验证目标，不允许“执行很多动作但无价值”：

- 发现能力：`candidate_opportunities_count`
- 质量能力：`edge_exec_positive_rate`
- 漏斗推进：`spec_ready_delta`, `forecast_ready_delta`
- 执行能力：`signal_to_trade_conversion`

### 4.3 验收双轨制

`系统健康验收` 与 `策略价值验收` 必须拆分：

- 健康验收：服务可用、任务执行、数据时效、故障可诊断
- 价值验收：是否发现可执行机会、是否形成正向收益

避免出现“系统 PASS 但价值 NOT PASS”却无法定位责任的情况。

### 4.4 失败分层归因

统一故障域：

- 配置失败（凭证/配额/开关）
- 数据失败（spec/city/forecast 缺失）
- 策略失败（无候选/阈值不合理）
- 执行失败（下单/风控/撮合）

每次失败必须落到唯一主因类别，支持自动告警与自动修复编排。

## 5. Prompt 工程（目标要求）

Prompt 必须确保：

1. 先读事实，再做决策（禁止猜测驱动）
2. 输出结构化 plan/action/validation
3. 每轮仅允许 1-2 项参数实验变更
4. 明确回滚条件（连续 N 轮无改善则回滚）
5. 输出 `no_op_reason`（说明为何不交易/不推进）

---

## 6. 记忆架构（目标要求）

### 6.1 MVP 记忆

- 轮次级结构化摘要（假设、动作、结果、结论）
- 下一轮默认注入最近 N 轮关键记忆

### 6.2 扩展能力

- 预留外部 memory adapter（向量/图/混合）
- 统一 memory contract，避免与具体实现耦合

### 6.3 记忆质量约束

- 记忆必须可追溯到 session/step/tool_call
- 记忆必须可验证（有指标变化证据）
- 记忆必须可衰减（过期/压缩）

---

## 7. 价值闭环（目标流程）

1. 采集：market/weather/chain
2. 规格化：city/spec/freshness 对齐
3. 评估：`edge_exec = P_est - P_exec - friction - uncertainty_penalty`
4. 决策：run/hold + no_op_reason
5. 执行：交易前风控硬约束
6. 复盘：写入 memory + validation
7. 进化：下一轮基于证据调参或回滚

---

## 8. 可观测性与验收口径（目标态）

### 8.1 健康侧

- job 成功率、错误码分布、连续失败
- 数据新鲜度
- 漏斗阻塞告警（spec/fetch/signal/trade）

### 8.2 价值侧

- 候选机会数
- 可执行 edge 占比
- 信号转交易率
- 净收益、胜率、回撤

### 8.3 诊断侧

- no_op 原因分布
- 参数实验效果变化（A/B）
- 失败分层统计（配置/数据/策略/执行）

---

## 9. 迁移与门槛

分阶段推进：

- Phase A：健康可观测齐备（先可诊断）
- Phase B：机会发现稳定（小时级）
- Phase C：收益验证稳定（天级）

切换门槛（示例）：

- 连续 7 天健康侧 SLO 达标
- 候选机会持续产出并可解释
- 价值侧指标不劣于基线策略

---

## 10. 结论

本架构合理，方向正确。  
关键不在“是否接入 AI”，而在“是否把 AI 放在控制层并形成证据驱动进化闭环”。
